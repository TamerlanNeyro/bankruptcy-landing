# BACKEND-PLAN.md — План бэкенда

Что реально нужно этому приложению на сервере. Ничего лишнего.

---

## Текущее состояние

Сейчас у приложения есть только один серверный обработчик:

- `POST /api/webhook` — принимает апдейты от Telegram (команды `/start`, `/help`), отвечает текстом с клавиатурой. Хранения данных не требует.

Весь контент сайта (юрист, услуги, вопросы квиза, реквизиты) — статика в `data.js`, правится вручную в коде, не через базу.

Форма заявки (`screens.form.submit()` в `app.js`) сейчас ничего никуда не отправляет — заглушка (`setTimeout` вместо `fetch`). Это единственное реальное отсутствующее звено.

---

## Что нужно хранить

Ровно одна сущность: **заявки** (заполненная форма записи на консультацию).

### Таблица `applications`

| Поле | Тип | Зачем |
|---|---|---|
| `id` | bigint, PK | Идентификатор записи |
| `name` | text | Имя из формы |
| `phone` | text | Телефон из формы — по нему юрист перезванивает |
| `time` | text | Удобное время связи (`утро`/`день`/`вечер`) |
| `source` | text | Откуда пришла заявка: `home` / `result` (после квиза) / `service` (с карточки услуги) — понимать, что лучше конвертит |
| `quiz_answers` | jsonb | Ответы квиза, если заявка после диагностики — юристу видно контекст ситуации клиента до звонка |
| `user_id` | bigint, nullable | Telegram ID пользователя — чтобы бот мог ему написать |
| `username` | text, nullable | Telegram username — быстро найти человека вручную |
| `status` | text, default `'new'` | `new` / `contacted` / `done` — минимальный трекинг, чтобы не перезвонить дважды или не забыть |
| `created_at` | timestamptz, default now() | Когда пришла заявка |

Больше ничего хранить не нужно: контент сайта — не данные приложения, а конфигурация кода; квиз не сохраняется отдельно от заявки (смысла в нём вне заявки нет — это не аналитика, а лишь входные данные для юриста).

## Какие API нужны

Всего два обработчика, оба уже частично покрыты структурой проекта:

| Метод | Путь | Кто вызывает | Что делает | Auth |
|---|---|---|---|---|
| `POST` | `/api/webhook` | Telegram (сервер Telegram) | Обрабатывает `/start`, `/help` | Не нужна (Telegram сам не проверяется по секрету — это уже так в проекте) |
| `POST` | `/api/submit` | Mini App (`app.js`, `fetch(contact.webhookUrl)`) | Валидирует payload, пишет строку в `applications`, (опционально) уведомляет юриста в Telegram | Не нужна отдельная — Supabase-ключ используется только на сервере, не в браузере |

**Ничего для чтения заявок городить не нужно.** У Supabase уже есть готовая Table Editor-панель — юрист смотрит заявки там же, где они лежат, без необходимости в отдельном админ-API/панели на сайте. Строить кастомный `GET /api/applications` + логин + UI ради одного пользователя (самого юриста) — это лишняя часть, которую потом придётся поддерживать.

**Опционально, но дёшево сделать:** после успешной записи в базу `/api/submit` может одним `fetch` дёрнуть `sendMessage` в Telegram Bot API и прислать юристу уведомление в личку с данными заявки — это заменяет необходимость постоянно проверять Supabase вручную. Не обязательно для MVP, добавляется в 10 строк when понадобится.

## Кто что видит и редактирует

| Кто | `applications` (заявки) | `data.js` (контент сайта) | `.env` (токены/ключи) |
|---|---|---|---|
| Клиент (пользователь Mini App) | Создаёт свою же заявку (insert), не видит чужие и свою же после отправки | Только читает (рендер сайта) | Нет доступа |
| Юрист (владелец) | Смотрит все заявки через Supabase Dashboard, вручную меняет `status` там же | Меняет через код (правки в `data.js` → коммит → деплой, делает Claude Code по запросу) | Не видит напрямую, знает что где лежит |
| Бот / сервер (`server.js`, `api/*.js`) | Пишет (insert-only) через сервисный ключ Supabase | Читает при сборке ответов (например, `APP_URL` в кнопках) | Читает из `.env` на сервере |
| Кто угодно из браузера | Нет прямого доступа к Supabase — весь путь идёт через `/api/submit` на своём сервере | — | — |

Важное архитектурное решение: **Supabase-ключ никогда не попадает в браузер.** Mini App шлёт данные на свой же `/api/submit` (как и сейчас задумано через `contact.webhookUrl`), а уже этот серверный обработчик обращается к Supabase с ключом из `.env`. Так исключается сценарий, где кто-то читает ключ в коде страницы и лезет в базу напрямую.

RLS (row level security) в Supabase стоит включить на `applications` без единой публичной policy — доступ только через `service_role`-ключ с сервера. Никаких анонимных чтений/записей напрямую из интернета.

## Что НЕ нужно (осознанно исключено)

- **Аутентификация/логин для юриста** — он один, доступ к Supabase Dashboard и так только у него по паролю аккаунта. Отдельная система логина на сайте — лишняя сущность.
- **Админ-панель на самом сайте** — Supabase Table Editor уже это закрывает.
- **CMS для контента (`data.js` из базы вместо кода)** — контент меняется редко, правки делаются через Claude Code за минуты. Переезд контента в БД + панель редактирования — это работа ради работы при текущем масштабе.
- **Хранение квиза отдельно от заявки / аналитика по квизу** — никто пока не просил отчёты «сколько людей прошли квиз, но не записались»; добавится, когда появится конкретный вопрос, на который это должно ответить.
- **Многопользовательская система (несколько юристов, роли)** — сейчас один юрист, один бот. Не проектируем под гипотетическое будущее.

## Реализация (кратко)

1. Таблица `applications` создаётся SQL-скриптом выше в Supabase SQL Editor.
2. Новый файл `api/submit.js` (тот же паттерн, что и `api/webhook.js`) — принимает POST, валидирует `name`+`phone`, делает `fetch` в Supabase REST (`POST {SUPABASE_URL}/rest/v1/applications` с заголовком `apikey`/`Authorization: Bearer {SUPABASE_SERVICE_KEY}`), возвращает `{ok:true}`.
3. `server.js` (VPS) добавляет маршрут `/api/submit` аналогично `/api/webhook` — переиспользует тот же файл, что и Vercel.
4. `.env` (и на VPS, и в Vercel) получает `SUPABASE_URL` и `SUPABASE_SERVICE_KEY`.
5. `data.js`: `contact.webhookUrl` = `/api/submit` (относительный путь — работает и на VPS, и на Vercel без хардкода домена).
6. `app.js` менять не нужно — `fetch(APP_DATA.contact.webhookUrl, ...)` уже реализован, просто сейчас закомментирован путь через пустой `webhookUrl`.

Никаких новых зависимостей (`npm install`) не требуется — Supabase REST API вызывается обычным `fetch`, без SDK.
